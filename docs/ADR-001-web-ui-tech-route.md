# ADR-001: Web UI 技术路线

**状态**: 已决定  
**日期**: 2026-04-12  
**决策者**: 项目维护者

## 背景

当前 Web UI 是纯 vanilla JS 单文件 (`app.js` ~3000 行)，无打包工具、无框架。
随着功能增长（定时任务、webhook、agent 管理、任务执行状态），继续在单文件中堆积会严重影响可维护性。

## 方案对比

### A: 保持 Vanilla — IIFE 模块化

- **优点**: 零构建步骤、部署简单、无依赖升级负担
- **缺点**: 无真正模块隔离、无 tree-shaking、调试靠全局搜索
- **适合**: 个人项目、功能稳定后不再大幅扩展

### B: Vite + Preact/Lit

- **优点**: 真正模块化、HMR 开发体验、组件复用、TypeScript
- **缺点**: 增加构建步骤、新增 devDependency、部署需要 build 产物
- **适合**: 持续功能扩展、团队协作

## 决定

**选择方案 A: 保持 Vanilla 路线。**

理由：
1. 个人部署项目，维护者单人，构建链越少越好
2. 当前功能集已接近稳定，后续新增主要是控制面视图（非高频交互组件）
3. OpenAI-compatible API (A10) 已打通，如需更强 UI 可直接用 Open WebUI / LobeChat 对接
4. 避免引入框架后 vendor/ 膨胀（当前 vendor/ 已有 3MB）

### 实施约束

- app.js 内用 `// ═══ SECTION ═══` 注释做逻辑分区，不做文件拆分
- 新增控制面视图（tasks/webhooks/executions）通过 modal 实现，复用现有 modal 模式
- 状态管理保持单一 `state` 对象，不引入新的全局变量

## 后续

如果未来需要更复杂的 UI（如拖拽工作流编辑器），再重新评估方案 B。
