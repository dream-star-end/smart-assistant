### 支付系统

#### [Medium] #M21 — `createPendingOrder` 首充并发绕过
- **文件**: `packages/commercial/src/payment/orders.ts:270-288`
- **问题**: `plan-10`(首充)只检查 `userHasAnyPaidOrder(uid)`,不检查 pending;也没有 `orders.plan_code` 列 + partial unique 约束。同一用户并发开两笔 plan-10 pending,两笔都能走到 `markOrderPaid`(只锁单个 order,不锁 user 维度)。当前仅 ¥10 业务价值低,但一旦 plan-10 加赠送就是实质损失。
- **建议(Codex 最小修)**:
  ```sql
  ALTER TABLE orders ADD COLUMN plan_code TEXT;
  CREATE UNIQUE INDEX uniq_first_topup_active ON orders(user_id)
    WHERE plan_code = 'plan-10' AND status IN ('pending','paid');
  ```
  `createPendingOrder` 对 plan-10 把 pending 也算占用;expired/canceled 后释放。
- **上线前阻塞?** 否(不阻塞),除非上线前准备给 plan-10 加赠送。

#### [Medium] #M22 — 支付回调未校验 `total_fee` 与 `appid`(Codex 独立发现)
- **文件**: `packages/commercial/src/http/payment.ts` `handleHupiCallback`
- **问题**: 验签通过后,`status=OD` 直接 `markOrderPaid`,**没有** 校验回调 `total_fee` 与本地 `orders.amount_cents` 一致,也没有校验 `appid === deps.hupijiaoConfig.appId`。签名能挡伪造,但不能替代业务字段一致性校验。若虎皮椒侧任何 bug 或同主体多 appid 混用,会把别的订单金额误标记到本地订单。
- **建议**: 在 `markOrderPaid` 之前(或其 tx 内)读取订单:
  ```ts
  if (String(callbackBody.appid) !== deps.hupijiaoConfig.appId) reject
  const callbackCents = Math.round(Number(callbackBody.total_fee) * 100)
  if (callbackCents !== current.amount_cents) reject + alert
  ```
- **上线前阻塞?** **建议阻塞**(真金白银路径,低成本高收益)。

#### [Info] 支付主路径 — clean
- `payment/hupijiao/sign.ts`:MD5 签名算法正确(按 key ASC 排序、跳过 sign/空值),`timingSafeEqual` 防 timing。
- `billing/ledger.ts`:`FOR UPDATE` 锁、BigInt 全程、`credit_ledger` RULE 保护,三个原子操作(debit/credit/adminAdjust)扎实。
- `adminAdjust` HTTP 层硬 cap |delta| ≤ ¥100 万 = 1e8 cents,tx 内 `balance+delta<0 → InsufficientCreditsError`。

### Auth 子系统

#### [Medium] #M1 — `handleConfirmPasswordReset` 无速率限制(argon2 CPU DoS)
- **文件**: `packages/commercial/src/http/handlers.ts:585-613` + `router.ts:367`
- **问题**: `handleRequestPasswordReset` 有 3/min/IP 限流 + turnstile;但 `handleConfirmPasswordReset` **没挂** `enforceRateLimit`,也没挂 turnstile。虽然 reset token 是 32 字节随机(暴破不现实),这条路径仍然会在 argon2id 上消耗 CPU(每次 ~64MiB × 3 轮)。而且 `hashPassword` 在 token lookup 之前就跑,找不到有效 token 也白付 CPU 代价。
- **建议**: 加 `enforceRateLimit(scope: "confirm_reset", windowSeconds: 60, max: 5)`,并把 `hashPassword` 移到 token lookup 之后。
- **上线前阻塞?** 否,但是低成本防御,值得上线前修。

#### [Medium] #M2 — `handleCheckVerification` 邮箱存在性枚举 oracle
- **文件**: `packages/commercial/src/http/handlers.ts:358-383` + `router.ts:365`
- **问题**: 注释声称"反枚举",但实际 `{verified: true/false}` 直接反映"该 email 且已验证"是否在 DB。配合 30/min/IP + botnet 可批量扫号。无 turnstile。
- **证据**:
  ```ts
  const result = await query(`SELECT email_verified FROM users WHERE email = $1`, [email])
  sendJson(res, 200, { verified: ... })   // ← true ⇔ 邮箱存在且已验证
  ```
- **建议**: 注册时下发短 TTL 的 server-side `pending_id`(HMAC 绑 email+nonce+exp 5min),轮询用 `pending_id` 不用 `email`;或要求 Turnstile + 5/min/IP + per-email Redis burst。
- **上线前阻塞?** 否,但上线后 1 周内应修。

#### [Medium] #M3 — `verifyCommercialJwtSync` 与 `verifyAccess` 不等价
- **文件**: `packages/commercial/src/auth/jwtSync.ts:29-74`
- **问题**: sync 版只校验 `alg==="HS256"`,不校验 `typ`、`crit`、`iat`、`jti`;注释里却写"等价"。用在两处关键路径:文件代理 v3Supervisor 调度、`BLOCKED_FOR_USER_RULES` 的 admin bypass。目前利用面有限,但后续 feature 演进容易误读注释。

- **上线前阻塞?** 否。

### node-agent Go(4113 行全栈)

#### [Critical] #C1 — `renew.HandleDeliver` 不验 x509 + reload 失败不回滚(Codex:保持 Critical,DoS 面被低估)🚫 **BLOCKER**
- **文件**: `node-agent/internal/renew/renew.go:162-203, 183-187`
- **问题**: 仅 `bytes.Contains(certPem, "-----BEGIN CERTIFICATE-----")` 字面量检查即 atomicWrite + ReloadTLS。**没有**:
  1. `x509.ParseCertificate`
  2. 签名链对到本地 `CACrt`
  3. cert 公钥与本地 `TLSKey` 匹配
  4. SAN URI == `spiffe://openclaude/host/<uuid>`

  master leaf 一旦被攻破,攻击者可向所有 node 分发任意 PEM(错 host cert / 过期 / 甚至非 CA 签)。**更糟**:`ReloadTLS` 因 key mismatch 失败 → 此时 cert **已写盘覆盖旧文件**,下次 agent 重启 ListenAndServe 直接 fatal。**一次投毒整片节点下线**。
- **Codex 修复(必修)**:
  ```go
  // 写盘前完整校验
  block, _ := pem.Decode([]byte(certPem))
  leaf, err := x509.ParseCertificate(block.Bytes)
  caPool := x509.NewCertPool(); caPool.AppendCertsFromPEM(caBytes)
  _, err = leaf.Verify(x509.VerifyOptions{Roots: caPool, CurrentTime: time.Now()})
  // SAN URI 精确匹配 spiffe://openclaude/host/<HostUUID>
  // tls.X509KeyPair(certPem, tls_key bytes) 必须 OK 证明 cert 与本地私钥匹配
  // 写入前保留 old cert,reload 失败 rollback old 并再次 reload
  ```

#### [High] #H7 — `signHostLeafCsr` 不校验 CSR 公钥算法/强度
- **文件**: `packages/commercial/src/compute-pool/certAuthority.ts:447-463, signLeafFromCaKey 377-398`
- **问题**: 直接把用户提供的 CSR 送 openssl x509 -req。openssl 默认接受任意 keyType/keySize(含 512-bit RSA)。被入侵 host 可提交弱密钥 CSR → master 签 → 降级攻击;还可借 renew 换整套 keypair 而旧私钥不失效。
- **最小修**: 签前校验 `EC prime256v1` 或 RSA ≥ 3072;renew 路径校验 CSR 公钥 == 该 host 当前 cert 公钥(拒 rekey);rekey 走 admin-initiated 路径。
- **上线前阻塞?** 否(需要前置入侵),但上线前建议修。

#### [Medium] #M4 — `remoteHosts` 无内网地址过滤(SSRF 面)

### 产品侧约束(2026-04-25 boss 明确)

> **"不能影响易用性,用户的使用体验要关注,要尽量让用户使用容器中的 agent 时能完成各种任务,对外也能正常访问。"**

这条约束重塑了修复方案取舍:**安全修复不能以阉割产品价值为代价**。v3 的卖点就是"容器里的 agent 能跑任意工具 + 对外访问"(浏览器 / MCP / dev server / 端口预览 / 远程 SSH 到用户自己的内网设备)。因此以下几条原本建议的硬化,**按产品价值降级或改方案**:

| 原建议 | 产品影响 | 决策 |
|---|---|---|
| **#H1** tunnel 只 allow `[18789]` | 用户 agent 启 Playwright dev server / htmlpreview / 自定义端口服务 → Web 前端 iframe 通过 tunnel 转发 `bound_ip:port` 预览。死白名单**直接废掉端口预览场景**。 | **改方案** → deny 清单(非 allow-only) |
